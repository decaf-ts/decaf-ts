# DECAF-10 — Backend Server Primitives (for-http/server) & Model Controller Builder/Factory

- **Status:** In Progress (broadened 2026-06-19; re-verified and extended with Audit finding #2 + TASK-179 on 2026-06-19 review)
- **Priority:** High
- **Goal:** Establish `for-http/server` as the single home for every framework-agnostic backend/server primitive (`@route` + verb decorators, `ControllerBuilder`/`RouteBuilder`, `DecafController`/`DecafModelController`, request transformers, the new `ModelControllerBuilder`/`ModelControllerFactory`), with `for-nest` reduced to a thin Nest-specific consumer of those primitives — so a future `@decaf-ts/for-express` (or any other framework adapter) can reuse them with minimal effort.

---

## 1. Overview
This spec started as "build a `DecafModelControllerBuilder` for `for-nest`." A 2026-06-19 audit (see "Audit finding" below) found that the originally-planned builder was implemented once with real test coverage but was **never committed** and the production code path (`for-nest/src/decaf-model/FromModelController.ts`) still hand-writes every CRUD/bulk/statement/query route directly on a hardcoded `DynamicModelController` class. Meanwhile, separate work already landed a first layer of generic, framework-agnostic primitives directly in `for-http/src/server/` — `route()` (the `@route` decorator), `ServerControllerBuilder`/`ServerMethodBuilder` (a generic, business-logic-free controller/route builder pair), `DecafController`/`DecafModelController`, `RequestContext`, and the `ServerRoute`/`RouteParam`/`RouteResponse` models — but none of it is wired into `for-nest` yet, no model-aware builder exists on top of it, and several pieces of it are incomplete stubs.

This spec is broadened to:
1.  Finish and harden what already exists in `for-http/server` (verb decorators, a real handler-carrying route builder, `@allowIf`/`@blockIf` hardening in `core`).
2.  Build the model-aware layer on top of it (`ModelControllerBuilder`, `ModelControllerFactory`) so it can reproduce — and exceed, via granular config — everything `for-nest`'s current hardcoded controller does.
3.  Relocate primitives that are framework-agnostic but currently sit in `for-nest` (request `Transformer`s) into `for-http/server`.
4.  Rewrite `for-nest` to consume these primitives instead of duplicating them: a dedicated `auth` module with its own `forRoot(...)`, and a `FromModelController` that delegates to `ModelControllerFactory`/`ModelControllerBuilder` instead of hand-writing routes.

**Audit finding #1 (2026-06-19):** `for-nest/workdocs/reports/coverage/lcov-report/src/decaf-model/{DecafModelControllerBuilder,query-routes}.ts.html` are stale coverage reports (dated 2026-03-21, 80% line coverage) for a `DecafModelControllerBuilder` class that matched this spec's original design almost exactly (`addRoute`, `addComplexQueries`, `build()`, a `registerRoute()` guard calling `isOperationBlocked(...)` before ever defining the method). `git log --all -- "*DecafModelControllerBuilder*"` in `for-nest` returns nothing — the file was never committed and no longer exists on disk. `FromModelController.create()` still hardcodes `class DynamicModelController extends BaseController { ... }` with every route written out by hand (`for-nest/src/decaf-model/FromModelController.ts:233`). Separately, `for-nest/src/decaf-model/decorators/ApiOperationFromModel.ts` defines its **own** local `isOperationBlocked(ctor, op)` (2-arg, CRUD-only) instead of importing the richer 3-arg `isOperationBlocked(ctor, kind, value)` that TASK-107 actually shipped in `core/src/utils/utils.ts` — so the statement/query-blocking capability TASK-107 built has no caller anywhere in `for-nest`. This spec's TASK-106/107 rows below are corrected to reflect this; the original "Completed" status for TASK-106 was wrong. *(Re-verified 2026-06-19 review: every file/line reference above and throughout sections 4.1-4.7 was checked directly against the working tree and confirmed accurate — including the exact stale-report dates and the empty `git log --all` result.)*

**Audit finding #2 (2026-06-19 review):** This spec's stated problem ("`for-nest` re-implements primitives that should live in `for-http/server`") already has a second, larger instance beyond the controller-builder gap: `for-nest/src/controllers.ts` defines its **own** `DecafController`/`DecafModelController` pair that independently duplicates — almost line-for-line — the `persistence()` fallback chain (`Service.get` → `ModelService.getService` → `Repository.forModel`) and the IP-header-parsing logic (`parseIpHeader`, `x-forwarded-for`/`x-real-ip`/`X-Forwarded-For`/`X-Real-IP`) that already exists, framework-agnostically, in `for-http/src/server/controllers/controllers.ts`'s `DecafController`/`DecafModelController`. The two class pairs are structurally parallel but not related by inheritance or import — `for-nest`'s `DecafRequestContext` (`for-nest/src/request/DecafRequestContext.ts`) does not extend `Context`/for-http's `RequestContext<REQUEST>` at all, it's a plain unrelated wrapper class. This duplication predates this spec, is not mentioned in sections 4.1-4.7 or the original task breakdown, and has real blast radius: `for-nest/src/controllers.ts`'s classes are consumed by `webhooks/controllers.ts`, `events-module/EventsController.ts`, `decaf-model/FromModelController.ts`, and `decaf-model/utils.ts`. Tracked as TASK-179 below; since `FromModelController` is a consumer of both this duplication and TASK-177's rewrite target, TASK-179 should land together with or immediately before TASK-177 to avoid migrating `FromModelController`'s base class twice.

## 2. Goals
*   [ ] Design a builder API (`addCreate`, `addStatement`, `addListBy`, `addComplexQueries`, etc.) that progressively assembles the controller prototype without embedding specific methods in the class definition. *(superseded by `ModelControllerBuilder`/`ModelControllerFactory` below — same intent, now framework-agnostic and living in `for-http/server`)*
*   [ ] Plug the builder into the existing controller factory so every model route honors the current persistence metadata and the `BlockOperations` flags (including the statement/query-key domain TASK-107 already shipped in `core` but that `for-nest` never calls).
*   [x] Expand the `BlockOperations` decorator/guard so it can also block prepared statements and query-key routes — shipped in `db-decorators`/`core` (TASK-107); not yet consumed by `for-nest` (tracked again below as part of TASK-177).
*   [ ] Finish the generic `@route` decorator family in `for-http/server`: add verb-specific decorators (`@get`/`@post`/`@put`/`@patch`/a fifth `DELETE`-verb decorator — **not** literally named `@delete`, see TASK-168's naming correction below) that each just call `route(verb, path)`.
*   [x] Harden `core`'s `@allowIf`/`@blockIf` so a `thisArg` missing a `logCtx` method fails with a clear `UnsupportedError` instead of an opaque `TypeError`; add the unit test coverage that currently does not exist at all.
*   [ ] Make `for-http/server` the canonical home for every backend/server primitive decaf-ts ships (routes, builders, controllers, transformers), with `for-nest` (and any future `for-express`, etc.) importing and composing them rather than re-implementing them.
*   [ ] Finish the generic `ServerControllerBuilder`/`ServerMethodBuilder` pair so it can actually carry and materialize a route's implementation (today `ServerMethodBuilder.withImplementation()` is a no-op and `ServerRoute` has no handler slot) — these two classes must remain pure builders with zero persistence/model-specific logic.
*   [ ] Build `ModelControllerBuilder` (in `for-http/server`, composing `ServerControllerBuilder`) exposing model-aware helpers (`addCreateRoute`, `addReadRoute`, bulk routes, statement/query routes, complex-query routes) capable of reproducing every route currently hand-written in `for-nest`'s `FromModelController`.
*   [ ] Build `ModelControllerFactory` (in `for-http/server`, using `ModelControllerBuilder`) that takes a per-model configuration object with granular toggles (`allowStatementlessQuery`, `allowGroupingQueries`, `allowBulkStatement`, etc.) and decides which routes to materialize, defaulting to today's for-nest behavior (everything enabled) for backward compatibility.
*   [ ] Make `ModelControllerFactory`'s route generation `@composed()`-aware: when a model's `@pk()` is `@composed()`, derive the route path from the composed `args`/`separator`, and when `filterEmpty` permits omitted parts, register the necessary fallback routes so every legal composed-key shape is reachable over HTTP. Drive statement/query coverage from `PreparedStatementKeys`/`PersistenceKeys` (the actual enums already in `core`) rather than ad hoc string literals.
*   [ ] Move the request `Transformer` primitives currently living in `for-nest` (`RamRequestTransformer`, `RequestToContextTransformer`) into `for-http/server`, since they are framework-agnostic; re-export from `for-nest` for compatibility.
*   [ ] In `for-nest`: consolidate all auth logic (`AuthInterceptor`, `DecafAuthHandler`, the `Auth()` decorator, `AUTH_META_KEY`/`AUTH_HANDLER` tokens) into its own module with a `forRoot(options)` entry point, configurable to control e.g. whether the interceptor is registered globally or per-controller.
*   [ ] In `for-nest`: rewrite `FromModelController` to delegate to `for-http/server`'s `ModelControllerFactory`/`ModelControllerBuilder` instead of hand-writing `DynamicModelController` — this is the "greatly improved" controller-builder mechanism the broadened spec is named for.
*   [ ] In `for-nest`: retire `for-nest/src/controllers.ts`'s duplicate `DecafController`/`DecafModelController` in favor of consuming `for-http/server`'s versions (Audit finding #2; tracked as TASK-179).
*   [ ] Verify the entire primitive stack end-to-end with a dedicated e2e test that defines complex test models (with relations and `@composed()` primary keys), boots a controller for each model under a different configuration expressed via model decoration, and overrides some of those per-model configs through the module-level configuration — proving both configuration layers (decoration-driven and module-driven) drive `ModelControllerFactory` route materialization correctly (tracked as TASK-185).

## 3. User Stories / Requirements
*   **US-1:** As a NestJS maintainer, I want controller constructors built through reusable builder steps, so I can ensure the generated surface remains consistent with metadata without hand-editing each route.
*   **US-2:** As a security reviewer, I need every model controller to automatically skip routes that are blocked by `@BlockOperations`, including CRUD operations, prepared statements, and dynamic queries.
*   **US-3 (new):** As a maintainer of a future framework adapter (e.g. `@decaf-ts/for-express`), I want every backend/server primitive (`@route`, builders, `DecafController`, transformers) to live in `for-http/server`, framework-agnostic, so I can reuse them instead of re-implementing `for-nest`'s logic.
*   **US-4 (new):** As an API designer, I want a single configuration object to control exactly which route classes (statements, grouping/aggregation queries, bulk operations) are exposed for a given model, without hand-editing generated controllers.
*   **US-5 (new):** As a developer using `@allowIf`/`@blockIf` on a class without a `logCtx` method, I want a clear `UnsupportedError` explaining the requirement, not an opaque crash.
*   **US-6 (new):** As a `for-nest` consumer, I want auth wiring isolated in its own module with a `forRoot(...)` so I can opt in/out of a global interceptor without touching unrelated controller code.
*   **Req-1:** The controller builder must never hardcode route methods; every endpoint is injected via builder helpers (`addCreate`, `addRead`, `addStatement`, `addComplexQueries`, etc.).
*   **Req-2:** The builder must trust the existing dynamic query generation and expose it via an `addComplexQueries`/`addComplexQueryRoute` helper so decorated repository queries continue to be wired into the controller surface.
*   **Req-3:** The `sqaggre` annotations/documentation on each controller should be left untouched, and any automation that currently inspects them must behave the same after the builder refactor.
*   **Req-4 (new):** `ServerControllerBuilder`/`ServerMethodBuilder` (the generic, framework-agnostic builders in `for-http/server`) must contain zero persistence/model-specific logic; all of that lives one layer up, in `ModelControllerBuilder`.
*   **Req-5 (new):** `ModelControllerFactory`'s configuration must support, at minimum: `allowStatementlessQuery: boolean` (expose query/complex-query routes directly, not only via the generic `statement` method), `allowGroupingQueries: boolean | <granular config>` (count/avg/max/min/sum/distinct/group routes), `allowBulkStatement: boolean | <granular config>` (bulk CRUD routes, with the ability to enable everything up to but excluding bulk update, or similar partial combinations).
*   **Req-6 (new):** Route generation must respect `@composed()` PK metadata: path segments derived from `composed.args`/`composed.separator`, with fallback route registration covering every combination `composed.filterEmpty` allows.
*   **Req-7 (new):** `@allowIf`/`@blockIf` must verify `typeof thisArg.logCtx === "function"` before calling it, throwing `UnsupportedError` (the existing class in `core/src/persistence/errors.ts`) when it is missing.
*   **Req-8 (new):** `for-nest`'s auth module must expose `forRoot(options)` returning a Nest dynamic module; `options` must include at least a flag controlling global-vs-opt-in interceptor registration.
*   **Req-9 (new, Audit finding #2):** `for-nest/src/controllers.ts`'s `DecafController`/`DecafModelController` must be retired in favor of `for-http/server`'s versions; any Nest-specific behavior they currently add (Express `Request` typing, `DecafRequestContext`/`DecafServerCtx`) must be reproduced by extending the generic classes, not by keeping a parallel implementation.
*   **Req-10 (new):** An end-to-end test MUST be added that defines test models complex enough to exercise the full controller-builder surface — at minimum: a single-PK model, a `@composed()` multi-part PK model (with `filterEmpty` enabled so fallback routes are required), and a model with `@relation()`/cross-model references (honoring the Cross-Relationship Guardrail from the constitution: exactly one side `populate: true`, the opposite side `populate: false` weak reference typed as `<pk_type> | <Class>`).
*   **Req-11 (new):** Each test model MUST have its controller booted under a distinct `ModelControllerFactory` configuration expressed by decorating the model class (per-model decoration-driven config) — e.g. one model allows grouping queries, another disables bulk update, a third allows statementless queries, a fourth blocks a prepared statement via `@BlockOperations` — so the e2e test proves the decoration → factory → route-materialization path for every config knob.
*   **Req-12 (new):** The e2e test MUST additionally override some of the per-model decoration-driven configurations through the module-level configuration (e.g. `DecafModuleOptions`/`controllerExposure` or an equivalent module-level config surface), proving the module config takes precedence over per-model decoration metadata where the two conflict.
*   **Req-13 (new):** The e2e test MUST assert, over live HTTP, that each model's exposed route surface matches its effective configuration (decoration-driven as overridden by module config) — blocked routes return 404 (or the configured "not exposed" status), allowed routes return the expected status codes — covering CRUD, bulk, statements, grouping/aggregation, complex queries, and composed-PK fallback routes for every legal `filterEmpty` shape.
*   **Req-14 (new):** The full API surface MUST be integration tested using a real `HttpAdapter` (`AxiosHttpAdapter` from `for-http`) as the client — no in-process/in-memory shortcut. Every operation class MUST be exercised over the wire: CRUD (create/read/update/delete), bulk operations, prepared-statement routes (e.g. `listBy`/`paginateBy`/`find`/`page`), grouping/aggregation queries (count/avg/max/min/sum/distinct/group), complex/dynamic queries, composed-PK routes (including `filterEmpty` fallback shapes), and server-emitted events (e.g. `/sse` delivery from the events module for each model operation).

## 4. Architecture & Design

### 4.1 `for-http/server` as the primitives layer
`for-http/src/server/` already contains the first generation of these primitives:
*   `decorators.ts` — `route(httpMethod, path)`, metadata-only, keyed by `ServerKeys.ROUTE`.
*   `constants.ts` — `ServerKeys.{CONTROLLER, ROUTE}`.
*   `types.ts` — `HttpVerbs`, `ServerRouteDecOptions`, `ServerModelRoute`, `ServerApiProperty`, `ServerParamProps`.
*   `controllers/controllers.ts` — `DecafController<REQUEST, RESPONSE, CONTEXT>` / `DecafModelController<M, REQUEST, RESPONSE, CONTEXT>`, both framework-agnostic (no Nest/Express import), already handling `logCtx`/persistence resolution/IP-aware logging.
*   `controllers/RequestContex.ts` — `RequestContext<REQUEST>` (framework-agnostic request context base).
*   `controllers/models.ts` — `RouteParam`, `RouteResponse`, `ServerRoute` (a `Model` describing a route's shape: method, path, params, body schema, responses, auth requirement — no handler).
*   `controllers/ControllerBuilder.ts` / `RouteBuilder.ts` — `ServerControllerBuilder` (abstract, `addMethod()`/`addMethodFromRoute()`, no concrete `build()` yet) and `ServerMethodBuilder` (fluent `withMethod/withPath/withPathParam/withQueryParam/withBodySchema/withResponse/...`).

Gaps to close (this spec's first wave of tasks):
*   No verb-specific decorators (`@get`, `@post`, ...) — only the generic `route()`. **Naming correction (2026-06-19 review, verified with `tsc`):** the fifth decorator cannot literally be named `delete` — it's a reserved JS/TS keyword, so `export function delete(path) {}` is a compile error (`TS1359`) and `@delete("/x")` as a decorator call would misparse as the `delete` operator. TASK-168 recommends capitalizing all five (`Get`/`Post`/`Put`/`Patch`/`Delete`) to match `@nestjs/common`'s own convention instead.
*   `ServerMethodBuilder.withImplementation()` is a no-op stub; `ServerRoute` has no field to carry an actual handler. The builder can describe a route's metadata but cannot yet attach behavior to it.
*   `ServerControllerBuilder.build()` is abstract with no concrete materialization logic anywhere — nothing currently turns accumulated `ServerRoute`s into a real class with real methods.
*   No model-aware layer (`ModelControllerBuilder`/`ModelControllerFactory`).
*   No request `Transformer` primitives yet (still in `for-nest`).

### 4.2 `core` auth hardening (`@allowIf`/`@blockIf`)
`core/src/auth/decorators.ts` proxies the decorated method and unconditionally calls `thisArg["logCtx"](args, target.name, true)`. There is no test coverage for either decorator today, and no guard for `thisArg.logCtx` being absent — calling either decorator on a class without a `logCtx` method currently throws an opaque `TypeError` from inside the proxy. Fix: check `typeof (thisArg as any).logCtx === "function"` before calling it and throw `UnsupportedError` (already used the same way for `maxConcurrentTransactions=0` in DECAF-7) with a message naming the decorator and the offending class/method.

**Additional finding (2026-06-19 review):** `allowIf` and `blockIf` (`core/src/auth/decorators.ts:12-58`) are, as of this review, **byte-for-byte identical implementations** apart from their function names — both call `handler(...)`, both throw whatever `AuthorizationError` the handler returns, both wrap a throwing handler in `InternalError`. Nothing today distinguishes "allow if" semantics from "block if" semantics at the decorator level; the distinction (if any) is left entirely to how a caller writes their `AuthHandler`. Confirmed via direct read and `grep` that there are zero current callers anywhere in the repo, so this has shipped without anyone exercising the difference. TASK-169 must get explicit confirmation from the spec owner on one of two paths before hardening: (a) this symmetry is intentional — the two names exist purely for call-site readability and both should keep matching behavior, the new tests should assert they behave identically — or (b) `blockIf` was meant to invert the condition (e.g. block when the handler does **not** return an error, or some other inverted contract) and this is a latent bug to fix alongside the `logCtx` hardening.

### 4.3 `ModelControllerBuilder`
Lives in `for-http/src/server/controllers/`, composes (not duplicates) `ServerControllerBuilder`. Exposes model-aware helpers mirroring what `for-nest/src/decaf-model/FromModelController.ts` currently hand-writes:
*   CRUD: `addCreateRoute`, `addReadRoute`, `addUpdateRoute`, `addDeleteRoute`.
*   Bulk: `addBulkCreateRoute`, `addBulkReadRoute`, `addBulkUpdateRoute`, `addBulkDeleteRoute`.
*   Statements: `addStatementRoute` (generic `statement/:method/*args`) plus the named shortcuts already hardcoded today (`listBy`, `paginateBy`, `find`, `page`, `findOneBy`, `findBy`).
*   Grouping/aggregation: `addGroupingQueryRoute` (covers `PreparedStatementKeys.{COUNT_OF,MAX_OF,MIN_OF,AVG_OF,SUM_OF,DISTINCT_OF,GROUP_OF}` — currently dead/commented-out code in `FromModelController.ts`).
*   Dynamic/complex queries: `addComplexQueryRoute` (today's `createQueryRoutesFromRepository`).
Each helper consults `isOperationBlocked(ModelConstr, kind, value)` (the real 3-arg `core` signature, not `for-nest`'s local 2-arg duplicate) before registering anything, so blocked routes are never materialized — not just hidden from documentation.

### 4.4 `ModelControllerFactory`
Also in `for-http/server`, built on `ModelControllerBuilder`. Signature roughly `create(ModelConstr, persistence, config?)`. `config` (all fields optional, defaulting to today's for-nest behavior — everything on):
*   `allowStatementlessQuery: boolean` — expose query/complex-query routes directly rather than only through the catch-all `statement` route.
*   `allowGroupingQueries: boolean | { count?, avg?, max?, min?, sum?, distinct?, group? }` — per-aggregation granularity.
*   `allowBulkStatement: boolean | { create?, read?, update?, delete? }` — e.g. enabling every bulk route except bulk update.
*   Composed-PK handling: when `Model.pk(ModelConstr)` resolves to a `@composed()` property, read its metadata (`DBKeys.COMPOSED` → `{ args, separator, filterEmpty, ... }`, the same metadata `FromModelController.getRouteParametersFromModel` already reads) to build the path (`:key1/:key2/...`) and register fallback routes for every combination `filterEmpty` allows to be omitted, so no legal composed-key shape 404s.
*   Use `PreparedStatementKeys` (`core/src/query/constants.ts`) and `PersistenceKeys.{QUERY,STATEMENT}` (`core/src/persistence/constants.ts`) as the source of truth for "what statement/query surface exists," instead of hardcoded strings, so new prepared statements automatically have a config knob.

### 4.5 Transformer relocation
`for-nest/src/ram/RamRequestTransformer.ts` and the `RequestToContextTransformer`/`requestToContextTransformer` it imports from `for-nest/src/interceptors/context.ts` are framework-agnostic (plain request → context shape mapping). Move both into `for-http/src/server/` (e.g. `transformers/`), re-export from `for-nest` so existing imports keep working, update internal `for-nest` references.

### 4.6 `for-nest` auth module
Today auth is split across `for-nest/src/decaf-model/decorators/decorators.ts` (`Auth()` decorator: `ApiBearerAuth()` + `UseInterceptors(AuthInterceptor)` + `SetMetadata`), `for-nest/src/interceptors/AuthInterceptor.ts`, and `for-nest/src/request/DecafAuthHandler.ts`. Consolidate all of it into a new `for-nest/src/auth/` module exposing `DecafAuthModule.forRoot(options)` (standard Nest dynamic-module pattern), where `options` includes at least `global: boolean` — `true` registers `AuthInterceptor` as an `APP_INTERCEPTOR` (every route protected by default), `false` keeps today's per-controller opt-in via `@Auth()`.

### 4.7 `for-nest`'s `FromModelController` rewrite
Once 4.3/4.4 exist, `FromModelController.create()` stops hand-defining `DynamicModelController` and instead: resolves persistence, calls `ModelControllerFactory.create(ModelConstr, persistence, config)` to get back route definitions/metadata, then layers the Nest-specific bits on top (`@Controller`, `@nestjs/swagger` decorators built from `ServerRoute`'s framework-agnostic `summary`/`description`/`responses`, `@Param`/`@Query`/`@Response` parameter decorators, `@Auth()`). This is the actual fulfillment of this spec's original ask (TASK-106) and the point where `@BlockOperations`'s statement/query targets (TASK-107) finally get a caller in `for-nest`.

### 4.8 Retire `for-nest`'s duplicate `DecafController`/`DecafModelController` (Audit finding #2)
`for-nest/src/controllers.ts` exports `DecafController<CONTEXT>`/`DecafModelController<M, C>`, independently reimplementing the same `persistence()` fallback chain and IP-aware logging that `for-http/src/server/controllers/controllers.ts` already provides framework-agnostically. Plan:
*   Make `for-nest`'s `DecafRequestContext` (`for-nest/src/request/DecafRequestContext.ts`) extend (or wrap, if a clean `extends` isn't viable given its current plain-class shape) for-http's `RequestContext<Request>` (Express's `Request` as the `REQUEST` type param), instead of being a structurally unrelated class.
*   Make `for-nest/src/controllers.ts`'s `DecafController`/`DecafModelController` extend for-http's versions, adding only what is genuinely Nest/Express-specific (if anything remains after the `RequestContext` unification above); otherwise replace the local exports with re-exports of the for-http classes.
*   Update all four current consumers (`webhooks/controllers.ts`, `events-module/EventsController.ts`, `decaf-model/FromModelController.ts`, `decaf-model/utils.ts`) to the unified base classes.
*   Sequence this with TASK-177 (`FromModelController` rewrite) since that file is a consumer of both efforts — migrating its base class once, not twice, is the point.

## 6. Open Questions / Risks
*   Are there existing `sqaggre` annotations tied to a static class definition that the builder must copy verbatim onto the generated prototype?
*   Do any modules rely on controller methods that currently exist but are not decorated as Nest routes (e.g., utility helpers)? The builder must keep those methods accessible even if they are not exposed over HTTP.
*   `ModelControllerFactory`'s config defaults must exactly reproduce today's for-nest route surface, or every existing consumer's API contract breaks silently on upgrade — TASK-172's parity tests are the safety net for this.
*   `ServerRoute`/`RouteResponse` (in `for-http/server/controllers/models.ts`) are framework-agnostic; confirm they carry enough information (status codes, schemas) for `for-nest`'s rewrite to still produce equivalent `@nestjs/swagger` decorators, or extend the models rather than reaching back into Nest-specific concepts.
*   Risk: `core/src/auth/decorators.ts`'s `allowIf`/`blockIf` are exported with zero current callers found in the repo outside their own module — confirm no downstream consumer depends on today's crash-on-missing-`logCtx` behavior before hardening it.
*   Risk: `allowIf`/`blockIf` are currently identical implementations (see §4.2 "Additional finding") — TASK-169 needs an explicit decision on whether that's intentional before writing tests that lock in the current (possibly accidental) behavior.
*   Risk: relocating Transformers and auth logic out of `for-nest` is a breaking change for any external consumer importing the old paths directly; decide whether to keep re-export shims permanently or only for one deprecation cycle.
*   Risk (Audit finding #2): unifying `for-nest/src/controllers.ts` with `for-http/server`'s `DecafController`/`DecafModelController` (TASK-179) touches 4 existing consumers (webhooks, events module, `FromModelController`, `decaf-model/utils.ts`) and for-nest's `DecafRequestContext`, which today is not related by inheritance to for-http's `RequestContext`. This is a larger, riskier refactor than any single task in the original breakdown and deserves its own regression pass, not just inclusion-by-reference in TASK-177/178.
